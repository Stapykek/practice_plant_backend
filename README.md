# Practice_Plant_Backend

Проект по написанию бекэнда для сервиса для по уходу за домашними растениями
Построен на основе микросервисной архитектуры.

## Что реализовано
* Сервис gate - API-шлюз, раскрывающий API-эндпоинты, с помощью которых
пользователь может взаимодействовать с сервером.
* Сервис auth-microservice - Регистрация и авторизация пользователей (пользователи 
могут иметь две роли: *стандартный* и *администратор*). После успешного логина
пользователь получает JWT (JSON Web Token) в котором хранятся свойства пользователя
(айди, имя и роль). JWT используется для авторизации и даёт пользователю
доступ к API-эндпоинтам соответствующим его роли.
* Сервис user-microservice - Операции с данными пользователей.
* Сервис plant-type-microservice - Операции с видами растений. Обычные пользователи могут 
лишь просматривать записи о видах растений в БД, а пользователи-администраторы
могут также добавлять, изменять и удалять эти записи. Есть возможность загрузки изображений
и прикрепления их к записи о виде растения. Загруженные файлы сохраняются на сервере и
любой пользователь может получить к ним доступ с помощью раздачи статических файлов.
* Сервис disease-microservice - Операции с типами заболеваний. Доступ к записям о типах
заболеваний осуществляется так же, как и в предыдущем сервисе.
* Сервис user-plant-microservice - Операции с растениями пользователей.
Любой пользователь может сохранить своё домашнее растение в базу данных. Также пользователи
могут загружать картинки для своих растений, доступ к ним происходит аналогично
сервису plant-type-microservice.
* Сервис disease-tag-microservice - Операции с отметками о болезнях растений.
Пользователи могут отметить заболевания своих домашних растений, чтобы упростить себе поиск
информации по уходу за ними.

**Такой набор микросервисов позволяет реализовать (с помощью фронтенда) пользовательские сценарии**:
### Регистрация и аутентификация
    [/auth/signUp] -> регистрация в сервисе
    [/auth/signIn] -> аутентификация, получение JWT для доступа к остальным сервисам

### Работа с записями о видах растений и типах заболеваний (администраторы)
    POST [/plant-type] -> добавление вида растений в БД
    PATCH [/plant-type] -> изменение информации о виде: добавление изображения, 
    описания, советов по уходу и т.д
    DELETE [/plant-type] или [/plant-type/image] -> Удаление вида растений из бд /
    Удаление изображения.
    (аналогично для [/disease], но без изображений)

### Добавление домашних растений пользователей
    POST [/user-plant] - добавление домашнего растения
    PATCH [/user-plant] - изменение информации/изображения домашнего растения,
    указание даты полива

### Просмотр информации о болезнях растений
    GET [/disease] или [/disease/many] - получение информаций о типах болезней
    растений

### Добавление/удаление отметок о болезнях растений
    POST [/disease-tag] - создание отметки, если растение заболело
    DELTETE [/disease-tag] - удаление отметки, если растение выздоровело

### Получение пользователем рекомендаций по уходу
    GET [/user-plant] или [/user-plant/many] - пользователь получает некоторое своё
    растение (там же хранится информация о его виде)
    GET [/plant-type] - пользователь просматривает информацию о виде растения, 
    получает рекомендации по уходу.
    GЕT [/disease-tag] - пользователь просматривает наличие заболеваний у своего
    растения (если они есть, получает ссылки на заболевания).
    GET [/disease] - пользователь просматривает заболевания растения и получает
    рекомендации по уходу за ним.

## Локальный запуск сервисов

1. **Установка зависимостей**

    Для этого нужно открыть терминал в корневой папке проекта
    и выполнить команду для установки зависимостей.
    ```shell
    npm install
    ```
2. **Запуск сервера NATS**
3. **Заполнение .env файла** 

    Пример заполненного .env файла: файл .example_env
4. **Настройка базы данных PostgreSQL**
    
    Нужно создать базу данных с таким же именем как в PG_DB_NAME в .env
    и с теми же данными для входа как в .env файле.
    
    Затем нужно выполнить скрипты schema:sync или 1) migration:generate, а затем 2) migration:run
    
    ```shell
    # синхронизация схемы бд с удалением старых данных
    npm run schema:sync
    
    # генерация и выполнение миграции схемы бд 
    # с сохранением старых данных 
    npm run migration:generate
    npm run migration:run
    
    ```
5. **Запуск нужных сервисов**

    Для этого используется следующая команда:
    ```shell
    nest start <service-name>
    ```
    *Прим.: аргумент "--watch" включит **горячий перезапуск**, т.е. изменения 
    будут применяться сразу же после сохранения файлов с кодом*
